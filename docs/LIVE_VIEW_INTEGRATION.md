# Живой экран агента Lumina — интеграция для маркетплейса

Покупатель в реальном времени смотрит, **как агент делает его заказ**: этапы, рассуждения
модели, действия инструментов, вердикты QA и превью готовых материалов. Хостинг, рендер и
нагрузка — целиком на стороне Lumina. Со стороны маркетплейса интеграция — **одна ссылка**:
никакого нового API, токенов, вебхуков или поллинга.

Базовый адрес сервиса (далее `<SERVICE>`):

```
https://lumina-marketplace-587790795280.us-central1.run.app
```

---

## TL;DR — вся интеграция в одну строку

На странице заказа у агента Lumina отрендерите кнопку/ссылку по шаблону из **вашего же**
`deliveryId` (тот, что вы присылаете нам в событии заказа; если его нет — `task.id`):

```html
<a href="<SERVICE>/live/d/{deliveryId}" target="_blank" rel="noopener">
  🧠 Смотреть, как агент делает заказ
</a>
```

Показывать можно сразу после нашего ответа `{"status": "accepted"}`. Если заказ ещё не дошёл
до нас, страница сама вежливо ждёт («Waiting for the agent to pick up this order…»), сама
обновляется и сама останавливается по завершении. Готово.

---

## Что видит покупатель

- **Этапы** производства: «👁️ Studying your product photo» → «🔎 Researching your brand» →
  «🎬 Art-directing the shoot» → «🎨 Producing your content» → «🔍 Quality control» →
  «📦 Packaging your delivery».
- **💭 Рассуждения модели** — краткие reasoning-сводки Gemini в момент принятия решений.
- **↳ Действия инструментов** человеческим языком: «rendering a lifestyle shot · 4:5»,
  «filming a product clip (9:16 · 8s)», «writing instagram copy in Russian».
- **Вердикты QA**: «✓ on-brand · 92%» / «✗ needs work — logo distorted» и перегенерации.
- **Превью** изображений/видео и итоговый scorecard качества — по мере готовности.

Страница строго read-only: ни кнопок, ни управления эскроу — её безопасно отдавать покупателю.
После завершения заказа она остаётся постоянным архивом «как это было сделано».

## Состояния страницы

| Состояние | Что показывается |
|---|---|
| Заказ ещё не дошёл до Lumina | Спиннер «Waiting for the agent to pick up this order…» (страница ждёт сама) |
| `InProgress` | Живая лента мыслей/действий, текущий этап, таймер |
| `Delivered` | Полная лента + галерея превью + QA-scorecard; поллинг остановлен |
| `Failed` | Лента до места ошибки + причина (`Failed: …`) |

## Четыре способа получить ссылку (по возрастанию усилий)

### Вариант 0 — ничего не делать (уже работает)

Ссылка уже вшита в `outputs.markdown` финальной доставки — последней строкой:

> 🧠 [Как создавался ваш пакет — журнал мыслей и работы агента](…)

Если ваш рендер markdown показывает ссылки, покупатель увидит её после доставки. Минус: только
*после* доставки — живой процесс он не застанет.

### Вариант 1 — шаблонная ссылка по deliveryId (рекомендуем)

Это TL;DR выше: `GET <SERVICE>/live/d/{deliveryId}` — обычная HTML-страница, без авторизации.
URL детерминированный, его можно зашить в шаблон страницы заказа и показывать мгновенно,
ничего не парся из наших ответов.

Повторная отправка ревизии (`task.revision_requested`) запускает новый прогон — этот же URL
автоматически указывает на самый свежий.

### Вариант 2 — поле `liveUrl` из наших ответов

Если удобнее брать готовый URL: мы возвращаем `liveUrl` в **синхронном ответе на заказ** и
дублируем его в **финальном callback**.

Ответ на `task.created` / `task.revision_requested`:

```json
{
  "status": "accepted",
  "liveUrl": "<SERVICE>/live/0f3a9c2d71b4",
  "note": "Watch the agent think & work live: <SERVICE>/live/0f3a9c2d71b4"
}
```

Финальный callback (оба исхода содержат `liveUrl`):

```json
{
  "status": "completed",
  "deliveryId": "…",
  "liveUrl": "<SERVICE>/live/0f3a9c2d71b4",
  "outputs": { "markdown": "…", "images": ["…"], "videos": ["…"], "cards": ["…"], "copy": {…}, "liveUrl": "…" }
}
```

Неизвестные поля можно игнорировать — контракт обратной совместим.

### Вариант 3 — iframe прямо на странице заказа

Страница встраивается без ограничений (нет `X-Frame-Options`, нет авторизации, адаптивная
вёрстка):

```html
<iframe src="<SERVICE>/live/d/{deliveryId}"
        style="width:100%;height:560px;border:0;border-radius:12px"
        loading="lazy"></iframe>
```

## Отладка

```bash
# жив ли резолвер deliveryId -> страница (404 = заказ ещё не дошёл до Lumina)
curl <SERVICE>/api/jobs/by-delivery/{deliveryId}
# -> {"job_id": "0f3a9c2d71b4"}

# сырое состояние работы (то, что поллит страница; обновляется каждые ~2.5 c)
curl <SERVICE>/api/jobs/{job_id}
```

## Безопасность и нагрузка

- Идентификатор работы — случайный токен: ссылка работает как capability link, списком нигде
  не отдаётся и не перечисляется.
- Страница и её API строго read-only; токенов, платёжных данных и контактов покупателя там нет.
  Держателю ссылки виден текст брифа (как и самому покупателю в заказе) — это его собственный заказ.
- Поллинг (раз в 2.5–3 с), трафик превью и хостинг страницы — на инфраструктуре Lumina
  (Cloud Run); на маркетплейс нагрузки нет.

## Чеклист внедрения

1. На странице заказа у агента Lumina отрендерить ссылку/кнопку
   `<SERVICE>/live/d/{deliveryId}` (вариант 1) — или взять `liveUrl` из ответа (вариант 2).
2. Всё. Остальное уже работает на нашей стороне.
